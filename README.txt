====================================================================
일정 자동화 보드 — 인수인계용 설명 문서
====================================================================
최종 업데이트: 2026-07-09

이 문서는 나중에 이 툴을 다른 사람에게 인수인계할 때, "이게 뭐고 어떻게
구성되어 있는지"를 빠르게 파악할 수 있도록 정리한 문서입니다.


--------------------------------------------------------------------
1. 이 툴이 뭐하는 물건인가
--------------------------------------------------------------------
FMV(풀모션비디오) 게임 여러 개를 동시에 진행하는 팀을 위한 일정 관리
웹앱입니다. 크게 세 가지 화면으로 구성됩니다.

  1) 프로젝트별 타겟 일정  — 각 프로젝트(PC/모바일)의 마스터 일정
     (킥오프, 촬영, 데모 심사, 런칭 등 고정된 마일스톤들)
  2) 팀원별 기획 로드맵    — 기획자 개개인이 담당하는 프로젝트별 세부
     기획 업무 목록 + 개인 업무량 캘린더
  3) 종합 캘린더 매트릭스  — 전체 프로젝트/기획자를 한 달력 위에서
     비교하거나, 누가 언제 비어있는지 보는 화면

데이터는 Firebase Firestore에 실시간 저장되어, 여러 명이 동시에 같은
화면을 봐도 서로의 수정 내용이 실시간으로 동기화됩니다.

구글 스프레드시트(마스터 일정 원본)에서 데이터를 직접 가져올 수 있고,
이때 기획자들에게 이미 배정된 업무의 날짜도 자동으로 최신화됩니다.


--------------------------------------------------------------------
2. 배포 방식
--------------------------------------------------------------------
정적 파일(HTML/CSS/JS)만으로 동작하며, GitHub Pages로 호스팅 중입니다.
서버 코드는 없습니다. 데이터 저장은 Firebase Firestore, 구글 시트
접근은 OAuth 로그인을 통해 브라우저에서 직접 호출합니다.

파일을 수정하면 이 6개 파일(index.html + css 1개 + js 4개)을 그대로
GitHub 저장소에 올려서 덮어쓰면 됩니다. 같은 이름이면 자동으로
덮어써집니다.


--------------------------------------------------------------------
3. 파일 구성
--------------------------------------------------------------------

■ index.html
  - 페이지의 뼈대(HTML 구조)와 스크립트 로딩 순서를 담당합니다.
  - Firebase SDK, Google Identity Services(OAuth 로그인용) 스크립트를
    <head>에서 불러옵니다.
  - Firebase 프로젝트 설정값(firebaseConfig)이 이 파일 안에 있습니다.
  - 화면 하단의 다크/라이트 모드 토글 버튼과 그 동작 스크립트도 여기에
    인라인으로 들어있습니다 (localStorage에 저장, 새로고침해도 유지됨).
  - 4개의 JS 파일을 아래 순서로 불러옵니다. **이 순서가 중요합니다**:
      state.js → sheets.js → render.js → main.js
    (뒤에 로드되는 파일이 앞의 파일에 정의된 함수/변수를 사용하기 때문)

■ style.css
  - 전체 디자인. 다크모드(기본)와 라이트모드 색상을 CSS 변수로 분리
    관리합니다 (:root 에 다크, body.light-theme 에 라이트 색상 정의).
  - 새 색상을 추가할 때는 반드시 두 군데(다크/라이트) 모두에 넣어야
    라이트모드 전환 시 글씨나 배경이 안 보이는 문제가 생기지 않습니다.
  - 겹침(빨간 배경)/지연(주황 배경) 업무 행 스타일, 모달/팝오버 애니
    메이션, 통계 카드 스타일 등이 이 파일에 있습니다.

■ state.js — 데이터와 핵심 로직 (화면을 직접 그리지 않음)
  주요 내용:
  - 전역 상태: tasks(전체 업무 배열), planners(기획자 배열), 공휴일
    데이터, 현재 보고 있는 화면(view) 등
  - Firebase Firestore 연결, 실시간 저장(persist)/구독(subscribeBoard)
    - 저장은 디바운스 처리되어 있어서, 여러 번 빠르게 수정해도 Firestore
      에는 마지막 상태 기준으로 한 번만 저장 요청이 나갑니다.
  - 날짜 계산 함수들 (근무일 계산, 공휴일/주말 판정, 다음 근무일 찾기 등)
  - PC/모바일 프로젝트별 "기획 업무 자동 템플릿 생성" 로직
    (buildPCTasks, buildMobileTasks, autoFillTemplate)
    - 마스터 일정의 특정 앵커(킥오프, 컨셉확정, 데모 심사, 런칭 등)를
      찾아서, 거기에 맞춰 세부 기획 업무들의 날짜를 자동으로 배치합니다.
    - 어떤 이름이 어떤 앵커로 인식되는지는 isKickoff, isConcept,
      isDemoJudgeQA 같은 판별 함수들에 정의되어 있습니다. 실제 시트의
      업무명이 바뀌면 이 판별 로직도 같이 확인해야 합니다.
  - 기획자 담당 가능 여부 확인(checkPlannerAvailability), 자동 배정
    알고리즘(computeAutoAssignPlan)
  - 마스터 앵커가 바뀌었을 때 기존에 생성된 기획자 업무를 다시 맞추는
    기능(resyncPlannerProjectAnchors)
  - 구글 시트/붙여넣기 데이터를 업무 목록으로 변환하는 파싱 로직
    (parseRows, parseTSVQuoted, parseAndAppend)
  - 겹침 감지(computeConflicts, findPlannerConflicts), 고정 앵커와
    겹치는 날짜로 수정하는 것을 막는 로직(findFixedAnchorOverlap)
  - 지연 판정(isDelayed): 완료 체크가 안 됐고 종료일이 지난 업무

■ sheets.js — 구글 시트 연동
  - 구글 OAuth 로그인 처리(초기화, 토큰 요청/갱신)
  - 실제 시트에서 데이터를 읽어오는 함수(fetchSheetTab)
  - "구글 시트에서 갱신" 버튼을 눌렀을 때 실행되는 전체 흐름
    (importFromGoogleSheets): 마스터 일정 교체 → 관련 기획자들 업무
    자동 재동기화 → 결과 모달 표시
  - 시트 ID, OAuth 클라이언트 ID, 가져올 탭 이름("PC 마일스톤",
    "M 마일스톤") 등 설정값이 이 파일 상단에 있습니다. 시트를 바꾸거나
    탭 이름이 바뀌면 여기를 수정하면 됩니다.
  - 주의: 이 시트는 "비공개"로 유지할 수 있도록 OAuth 로그인 방식을
    쓰고 있습니다 (API 키 방식이 아님). 로그인한 사람이 실제로 그
    시트를 볼 권한이 있어야만 데이터를 가져올 수 있습니다.

■ render.js — 화면을 그리는 함수들 (가장 큰 파일)
  - 세 개 화면(프로젝트별/기획자별/종합 캘린더) 각각을 그리는 함수
    (renderProjectView, renderPlannerView, renderOverviewView)
  - 업무 한 줄(행)을 그리는 함수 taskRow(마스터 일정용),
    plannerTaskRow(기획자 개인 업무용) — 겹침 아이콘(!), 지연 아이콘
    (⏰), 메모 아이콘(📝), 완료 체크박스 등이 여기서 조립됩니다.
  - 각종 모달(담당자 배정, 지원 인력 설정, 자동 배정 결과, 구글 시트
    갱신 결과 등)
  - 월간 달력 그리기(renderMonthGrid), 여유 기획자 찾기 캘린더
    (renderFreePlannerGrid)
  - 대시보드의 "이번 주 마감 요약" 패널도 renderProjectView 안에 있음
  - 참고: 프로젝트별 일정(마스터 일정) 화면에는 겹침 경고를 표시하지
    않습니다. 여러 트랙(타코-글비/영상/제작실)이 동시에 진행되는 게
    정상이라 일부러 뺐습니다. 기획자별 일정에는 겹침 감지가 그대로
    적용됩니다.

■ main.js (가장 작은 파일)
  - 사이드바 버튼들, 상단 네비게이션 클릭 이벤트 연결
  - 페이지가 열릴 때 처음 실행되는 코드(subscribeBoard, rerender)


--------------------------------------------------------------------
4. 데이터 구조 요약
--------------------------------------------------------------------
업무(task) 객체의 주요 필드:
  id, project(프로젝트 코드), category(분류), name(업무명),
  start/end(시작·종료일), workingDays(근무일수), progress(진행률%),
  locked(마스터 앵커 여부), fixed(날짜 수정 불가 여부),
  done(완료 체크), memo(메모), ownerPlanner(담당 기획자 id),
  supporters(지원 기획자 id 배열), rowGroup(같은 시트 행에서 나온
  항목끼리 묶는 용도, 서로 겹침 검사 안 함)

기획자(planner) 객체: id, name, projects(담당 프로젝트 코드 배열)

Firestore에는 board/shared 문서 하나에 tasks, planners, 공휴일 데이터,
완료 처리된 프로젝트 목록이 통째로 저장됩니다. (참고: 나중에 사용자가
많아지고 동시 편집이 잦아지면, 문서 하나에 다 몰아넣는 지금 구조는
"거의 동시에 수정하면 한쪽이 덮어써질 수 있는" 위험이 있습니다. 지금
규모에서는 문제없지만, 나중에 커지면 문서를 여러 개로 쪼개는 걸 고려
해볼 것.)

5. 꼭 알아야 할 것 / 주의사항
--------------------------------------------------------------------
- Firebase Firestore 보안 규칙 만료일: 2026년 8월 7일. 이 전에 규칙을
  갱신하지 않으면 그날부터 전체 접속이 막힙니다.
- 구글 시트 API 키/OAuth 클라이언트 ID가 sheets.js에 평문으로 들어가
  있습니다. OAuth 클라이언트 ID는 "승인된 자바스크립트 원본"으로 실제
  배포 주소만 등록해뒀기 때문에 다른 사이트에서는 악용할 수 없습니다.
- 이 프로젝트는 Node.js 빌드 과정이 없습니다. 파일을 그대로 열면
  바로 동작합니다 (단, Firebase/구글 로그인은 인터넷 연결 필요).
- 로컬에서 파일을 더블클릭해서 열면(file:// 주소) 구글 시트 연동이
  차단됩니다. 반드시 실제 배포 주소(GitHub Pages 등)에서 테스트해야
  합니다.
- 마스터 일정 앵커 이름이 실제 시트에서 바뀌면(예: "UI 투입일" →
  다른 이름), state.js의 isKickoff/isConcept/isDemoJudgeQA 등 판별
  함수들도 같이 확인해야 자동 배치 로직이 계속 정상 작동합니다.
